# 玩家状态统一方案

## 1. 背景

当前项目在玩家主状态上同时存在两套入口：

- `GameState.county_data`
- `GameState.player_unit.unit_data`

模型层已经通过 `GameState.get_unit_data()` 表达了“玩家当前治理单元的数据应优先来自 `player_unit.unit_data`”这一方向，但业务代码中仍大量直接访问 `game.county_data`。

这会导致以下问题：

- 县级与知府级代码对“当前状态”的理解不一致
- 某处修改后，另一处从另一份状态读取，表现为“修改丢失”
- 新增更高层级（省、朝廷）时，`county_data` 继续被误用
- JSONField 的原地修改问题与“双状态源”问题相互叠加，排障困难

## 2. 目标

本次改造的目标不是立即移除 `county_data`，而是先建立清晰的状态访问约定：

1. 统一“当前玩家状态”的唯一真相源
2. 统一状态的读取、修改、保存入口
3. 为后续逐步替换 `game.county_data` 的直接访问提供基础设施
4. 在不破坏现有前端契约的前提下，降低状态分叉风险

## 3. 核心决策

### 3.1 真相源选择

- 当 `game.player_unit_id` 存在时，**当前玩家状态以 `game.player_unit.unit_data` 为准**
- 当 `game.player_unit_id` 不存在时，回退到 `game.county_data`

### 3.2 `county_data` 的新定位

`county_data` 在本阶段不删除，但降级为：

- 县级旧存档兼容入口
- 县级接口返回的兼容镜像
- 创建新县级游戏时的初始化中转字段

换言之：**县级游戏可以继续返回 `county_data` 给前端，但业务逻辑不再默认把它当唯一主状态直接读写。**

### 3.3 保存策略

状态保存统一采用“整份替换”的模式，不依赖嵌套 dict 原地变更被正确持久化。

## 4. 方案设计

## 4.1 新增状态访问层

新增文件：`backend/game/services/state.py`

提供以下能力：

- `load_player_state(game, refresh=False) -> dict`
  - 返回当前玩家状态的深拷贝
  - 默认从 canonical source 读取

- `save_player_state(game, state, mirror_legacy=True)`
  - 将整份状态写回 canonical source
  - 若玩家当前单元是县级，则同步镜像到 `game.county_data`

- `mutate_player_state(game, mutator, mirror_legacy=True) -> dict`
  - 统一“读取 → 修改 → 保存”流程
  - `mutator(state)` 只操作内存中的 state

- `load_county_state(game, refresh=False) -> dict`
  - 面向县级逻辑的便捷入口
  - 内部仍然走 canonical source

## 4.2 读写约定

### 允许

- 通过状态访问层读取当前玩家状态
- 通过状态访问层保存当前玩家状态
- 在极少数兼容场景读取 `game.county_data`，但需要明确说明原因

### 禁止

- 在新的业务代码中直接写 `game.county_data[...] = ...`
- 在新的业务代码中直接把 `game.county_data` 视为县级玩家的唯一状态源
- 依赖深层 JSON 原地修改后自动在多处对象上保持一致

## 4.3 API 契约兼容

短期内不改变县级详情接口中的 `county_data` 字段名。

实现策略：

- Serializer 内部从 canonical state 读取
- 对外仍然组装为 `county_data`

这样可以避免前端在同一轮改造中被迫大规模修改。

## 5. 第一批执行范围

### 5.1 基础设施

- 新增 `backend/game/services/state.py`
- 在 `services/__init__.py` 中导出必要方法

### 5.2 首批高风险接入点

优先替换以下路径：

1. `GameDetailSerializer`
   - `county_data`
   - `available_investments`
   - `disaster_relief_advice`

2. `views.py` 中直接改 `game.county_data` 的核心县级入口
   - 新建游戏后的玩家 profile flavor 写入
   - 土地勘查请求
   - 其他明显的县级直接写入口

3. 少量高频服务
   - 先从与界面主链路直接相关的服务开始
   - 暂不一口气替换所有服务层访问

## 6. 执行顺序

### Phase 1：建立访问层

- 新增状态访问工具
- 添加县级/知府级/旧存档三种场景的最小测试

### Phase 2：切主链路读取

- 县级详情接口改走 canonical state
- 相关 serializer 改走统一入口

### Phase 3：切主链路写入

- 替换高风险视图中的直接 `game.county_data` 写入
- 将“读-改-写”收口到 `mutate_player_state`

### Phase 4：清理服务层

- 用 `rg "game\.county_data" backend/game` 盘点剩余直接访问
- 分批替换高频路径

## 7. 测试策略

至少补以下测试：

1. 县级游戏：修改 canonical state 后，`player_unit.unit_data` 与 `game.county_data` 保持一致
2. 知府游戏：状态修改只落在 `player_unit.unit_data`
3. 旧存档：没有 `player_unit` 时仍能回退到 `county_data`
4. `GameDetailSerializer` 返回的是 canonical state，而不是裸读 `GameState.county_data`

## 8. 非目标

本轮不做以下事情：

- 不移除 `GameState.county_data`
- 不把 `treasury`、`morale` 等字段迁移成独立数据库列
- 不一次性替换全项目所有 `game.county_data` 访问
- 不重写前端状态结构

## 9. 风险与回滚

### 风险

- 县级旧逻辑可能仍假设 `game.county_data` 是主状态源
- 某些 service 仍混用两套入口，导致局部行为暂未完全收口
- serializer 与 view 的切换如果不成对，可能导致显示和保存来源不一致

### 回滚策略

- 访问层是增量引入，若某个接入点有问题，可单独回退该接入点
- 第一批改造尽量保持接口返回结构不变，降低回滚成本

## 10. 本轮完成标准

本轮执行完成后，应满足：

- 已有一个明确的 canonical state access layer
- 县级详情接口不再直接裸读 `game.county_data`
- 至少一批县级写路径改走统一写入入口
- 有测试覆盖县级 / 知府 / 旧存档三类核心状态读取保存行为
